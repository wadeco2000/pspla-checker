# PSPLA Checker — Full Project Context for Claude Code

## What This Project Does
Automated tool that finds NZ security companies on the internet, checks whether each one holds a current PSPLA (Private Security Personnel Licensing Authority) licence, and stores results in Supabase. A Flask dashboard lets the user browse results, trigger searches, review AI decisions, and correct errors. A public GitHub Pages site shows a filtered read-only view encrypted with StatiCrypt.

Owned and operated by Wade. The goal is to build a comprehensive list of every NZ private security company and whether they are licenced.

---

## Tech Stack
- **Python** — Flask dashboard + searcher engine
- **Supabase** — PostgreSQL database, accessed via REST API with `apikey` header
- **Anthropic Claude API** — Haiku for fast tasks, Sonnet for deep verification
- **SerpAPI** — Google search wrapper (all web searches go through this, paid per query)
- **GitHub Actions** — scheduled searches (full, weekly, facebook, directories) run via cron workflows
- **Azure App Service** — hosts the Flask dashboard (Linux container)
- **BeautifulSoup** — HTML scraping
- **StatiCrypt** — AES-256 client-side encryption for public GitHub Pages site
- **Dropbox API** — automatic CSV backups to Dropbox on backup and clear-db

## Supabase Security (RLS)

Both tables have Row Level Security enabled:
- **Companies**: `public_read_only` policy — anon role can SELECT, nothing else. Service role bypasses RLS entirely.
- **AuditLog**: RLS enabled with NO policies — fully blocked to anon. Only service role can read/write.

This means the public site (using anon key) can read Companies but cannot see AuditLog or write anything.

## Full Search Pipeline (process_and_save_company)

Every company found — whether via Google, Facebook, or a directory — passes through the same pipeline in `process_and_save_company()`:

### 1. Dedup check
- `get_company_by_name()` — if company name already in DB, just append the new region and return
- `company_name_exists()` — final dedup guard at the end before save

### 2. Google Business Profile — `get_google_business_profile(company_name, region)`
One targeted SerpAPI search per company. Parses `knowledge_graph` and `local_results` from SerpAPI JSON — returned alongside organic results at no extra quota cost. Extracts rating, reviews, phone, address. Stored in `google_rating`, `google_reviews`, `google_phone`, `google_address`.

### 3. Facebook page scrape (if `facebook_url` is known)
`scrape_facebook_page(fb_url)` — three-tier approach:
- **Tier 1**: parse Google/SerpAPI snippet from `_FB_SNIPPET_CACHE`
- **Tier 2**: stream first 8 KB of page HEAD to extract `og:description` meta tag
- **Tier 3**: `m.facebook.com/slug/about/` with iPhone UA + 1s delay
FB email backfilled to company email if none found elsewhere.

### 4. PSPLA check — `check_pspla(company_name, website_region, page_text, co_result, directors, extra_context)`
Four strategies tried in order, stopping at first hit:

**Strategy 1 — Name variations:** `generate_name_variations()` — original, hyphen→space, CamelCase split, compound word split. Prefers Active licence; falls back to expired.

**Strategy 2 — Keyword AND search:** `extract_keywords()` strips stop words and joins first 3 keywords with AND.

**Strategy 3 — Single keyword:** First meaningful keyword with 6+ chars.

**Strategy 4 — LLM-suggested names:** Claude Haiku given ALL available context and asked to suggest up to 5 PSPLA search terms. Writes `llm_decision` audit entry.

**After finding results:**
- Multiple companies OR keyword/variant match → `verify_pspla_match()` called
- Single company + full-name match → trusted without verification
- Region-boosted sorting applied before verification
- Best-status doc selected: Active > Expired > Withdrawn, then most recent expiry

### 5. PSPLA verification — `verify_pspla_match(website_company, pspla_company, website_region, pspla_address, _audit_name)`

**Step A — Hard pre-check (no LLM):**
1. Forward check: every significant word (4+ chars) from website name must appear in PSPLA name. If ALL missing → reject.
2. **Reverse prefix check (added 10 Mar 2026):** if the PSPLA name starts with distinctive word(s) not present in the website name, it's a different entity. E.g. website="Livewire Electrical", PSPLA="Addz Livewire Electrical" → "addz" is a distinctive prefix → reject. Geographic qualifiers (southern, central, northern etc.) are exempt.

**Step B — Lessons injection:** relevant lessons from `lessons.json` injected into prompt.

**Step C — Claude Sonnet verify:** Returns `{match, confidence, reason}`. Writes to AuditLog.

**Step D — Deep verify:** If low/medium confidence + extra context available → `_llm_deep_verify()`.

**Graceful degradation:** If Claude API unavailable, returns low-confidence pass rather than false.

### 6. Companies Office check — `check_companies_office(company_name)`
Google search `site:companies.govt.nz`. Extracts registered name, number, NZBN, address, directors, CO status, incorporation date. Sold/re-registered detection: if status=Removed → broader keyword search for successor → retries PSPLA with successor name.

### 7. NZSA check — `check_nzsa(company_name, website=None)`
Scrapes `security.org.nz` member directory. Returns member, member_name, accredited, grade, contact_name, phone, email, overview.

**Matching logic (updated 10 Mar 2026):**
- Exact normalised name match → immediate accept
- Website domain match → immediate accept
- Word overlap scoring: significant word hits × 3, generic word hits × 1. Threshold: score ≥ 3.
- `_GENERIC` stop-words include: security, services, alarm, electrical, plumbing, construction, engineering, contracting, etc. Generic industry words alone do NOT constitute a significant match.
- **Domain mismatch guard:** if company website domain and NZSA member email/website domain are both company-specific (not Gmail/Hotmail etc.) and not in the directory-site exclusion list, and they differ → reject. Catches "Livewire Electrical" matching "Sefton Electrical" via shared generic word.
- **Directory domain exclusion:** if company's stored URL is a directory site (moneyhub.co.nz, yellowpages.co.nz, facebook.com etc.) skip domain mismatch check — not the company's own domain.

### 8. Cross-check — `_llm_cross_check_sources()`
Claude Haiku cross-checks all source names/addresses for consistency. Informational only.

### 9. Service detection — `gather_service_text()` + `detect_services()`
Fetches root domain, extracts internal links matching service slugs, scrapes up to 3 sub-pages. Claude Haiku detects `has_alarm_systems`, `has_cctv_cameras`, `has_alarm_monitoring`. Also runs on Facebook data → `fb_alarm_systems`, `fb_cctv_cameras`, `fb_alarm_monitoring`.

### 10. LLM association review — `llm_verify_associations()`
Called before saving. Rejects bad LinkedIn/FB/PSPLA associations, writes audit entries.

### 11. Individual licence check
Searches PSPLA for each director name. Sets `individual_license_found` and `license_type`.

### 12. Save — `save_to_supabase(record)`
HTTP POST/PATCH to Supabase REST API. Uses `RECORD_TEMPLATE` as base dict.

## The Corrections System

**Interactive correction flow (updated 10 Mar 2026):**
1. User types a correction note in the dashboard correction form
2. `POST /save-correction` — parses note, runs `check_pspla` to get a proposed new match, generates a lesson. Returns `{proposed: {...}}` WITHOUT saving to DB yet.
3. Dashboard shows a **Recheck Review Modal** with the proposed match details
4. User clicks "Yes, use this match" or "No, not a match"
5. `POST /confirm-recheck` — saves the approved/rejected result to DB

**Corrections files:**
- `corrections.json` — list of `{type, blocked_pspla_name, company_name, company_id, raw, summary, timestamp}`
- `lessons.json` — list of `{pattern_name, what_went_wrong, rule_to_apply, keywords_to_watch}`

`_is_pspla_match_blocked(company_name, pspla_name)` checked inside `check_pspla` before any match is accepted.

## Key Design Patterns

- **`RECORD_TEMPLATE`** — single source of truth for all DB columns. Every record starts as a copy.
- **`check_schema()`** — runs at startup of every search script. Verifies all RECORD_TEMPLATE keys exist. Aborts if missing.
- **`running.flag` / `pause.flag`** — file-based IPC. Cleared on dashboard startup and stop.
- **`search_start.json`** — written by `_launch()` at search start. Contains `{started, type, triggered_by}`. Used by `/search-running-info` for conflict detection.
- **`__main__` guards** on all `run_*.py` — prevents accidental execution when imported.
- **Orphan process kill** — Stop uses PowerShell WMI to kill python processes by script name.
- **`_FB_SNIPPET_CACHE`** — module-level dict, populated by `find_facebook_url`, consumed by `scrape_facebook_page`.
- **Sold company detection** — CO status=Removed → successor keyword search → retry PSPLA.
- **Region-boosted PSPLA sorting** — PSPLA names containing website region words ranked first.
- **`_LoggingAnthropicClient`** — intercepts all LLM calls, logs to `llm_debug.log`.
- **CSV exports** — always use `sorted(companies[0].keys())` for columns, never a hardcoded list.

## NZSA Report — Server-Side Form Submission (added 12 Mar 2026)

The "Report to NZSA" button on expanded company detail cards submits the NZSA Gravity Forms report form server-side via `requests` (replaced previous Playwright approach which required a visible browser, broken on Azure).

**Flow:**
1. Dashboard modal pre-fills sender details (defaults: Anon/Anon, anon@anon.co.nz), company info, and auto-generated evidence text
2. `POST /open-nzsa-report` — calls `_nzsa_build_payload()` which GETs the live form page, extracts nonces/hidden fields via BeautifulSoup, builds the POST payload
3. Supports `dry_run` mode — returns payload without submitting (for testing)
4. Required field validation in JS (`_nzsaValidate()`) — checks fname, lname, email (with @ check), company name, evidence

**Key details:**
- Form URL: `https://security.org.nz/public-info/report-unlicensed-operators/`
- Gravity Forms ID: 13
- Fields: `input_2.3` (fname), `input_2.6` (lname), `input_3` (email), `input_4` (mobile), `input_6` (party_name), `input_7` (location), `input_8` (evidence)
- Nonces extracted fresh each submission from the live page

## LinkedIn Overseas Rejection (added 12 Mar 2026)

Three-layer protection against accepting overseas LinkedIn profiles for NZ companies:

**Layer 1 — Search scoring penalties** (`find_linkedin_url()` in `searcher.py`):
- `_HARD_OVERSEAS_LI` list: Australia, UK, USA, Canada, Ireland, South Africa, India, Singapore keywords
- `_NZ_SIGNALS_LI` list: NZ city names, .co.nz domains
- Overseas penalty: −10 points; NZ signal bonus: +3 points

**Layer 2 — Hard location gate** (after `scrape_linkedin_page()`):
- If scraped `location` contains any overseas keyword → reject immediately, clear `linkedin_url`

**Layer 3 — AI verification** (`llm_verify_associations()`):
- Now accepts `linkedin_location`, `linkedin_industry`, `linkedin_description`, `linkedin_website` params
- Prompt rule: "If the LinkedIn location shows a different country… then REJECT"
- Also checks website domain mismatch between LinkedIn and company

**Dashboard `/find-linkedin` endpoint** also applies layers 2+3 for individual rechecks.

## Security Hardening (added 13 Mar 2026)

Ten-point security review implemented across the dashboard:

1. **HTTP security headers** — `Strict-Transport-Security`, `X-Content-Type-Options`, `X-Frame-Options`, `Referrer-Policy`, `Permissions-Policy` set via `@app.after_request`
2. **Session cookies** — `SESSION_COOKIE_SECURE`, `SESSION_COOKIE_HTTPONLY`, `SESSION_COOKIE_SAMESITE='Lax'`
3. **CSRF protection** — meta tag `<meta name="csrf-token">` + JS fetch interceptor auto-attaches token to all POST/PUT/DELETE requests. Server validates `X-CSRF-Token` header or `_csrf_token` form field via `@app.before_request`
4. **Rate limiting** — in-memory counters with thread locks on login, API endpoints. Configurable per-route limits
5. **Input validation** — company name length limits, email format checks, sanitised error responses
6. **Error handlers** — custom 404/500 handlers return safe HTML, no stack traces in production
7. **Admin guards** — `_is_admin()` check on all sensitive endpoints (clear-db, export, user management, rollback, etc.)
8. **CORS** — not needed (same-origin only), no CORS headers set
9. **Secrets management** — all secrets in `.env`, never logged or exposed in responses
10. **Audit logging** — every DB change writes to Supabase `AuditLog` table with timestamp, action, changes, triggered_by

## Crash Notification Email (added 13 Mar 2026)

When a search process crashes (exits without writing a completion record), the system:
1. `_detect_and_mark_crashes()` runs on dashboard startup — checks for stale `running` entries in `search_history.json`
2. Reads `search_start.json` to determine the search type and start time
3. Spawns a background thread to send `_send_crash_notification_email()` — red-themed HTML email to `NOTIFY_EMAIL`
4. Marks the history entry as `status: "crashed"` with a note

**Root cause fixes also applied:**
- Fixed `__time.time()` typo (double underscore) → `_time.time()` in `_search_process_alive()`
- Added double-launch guard in `_launch()` — blocks if `running.flag` is < 5 seconds old

## Common Tasks for Claude

**Add a new Supabase column:**
1. Add key to `RECORD_TEMPLATE` in `searcher.py` with `None` default
2. Populate in the record dict inside `process_and_save_company`
3. Add to detail row HTML in `dashboard.py`
4. User adds the column in Supabase manually
5. `check_schema()` auto-detects it

**Debug a specific company:**
```python
from dotenv import load_dotenv; load_dotenv('.env')
from searcher import check_pspla, check_companies_office, check_nzsa
co = check_companies_office("Company Name")
pspla = check_pspla("Company Name", website_region="Auckland", co_result=co)
nzsa = check_nzsa("Company Name", website="https://example.co.nz")
```

**Test NZSA matching on existing DB records:**
Write a small script importing `check_nzsa` and fetching records with `nzsa_member=eq.true` from Supabase. Compare old vs new results.

**If LLM keeps rejecting good matches:**
Check `lessons.json` for overly aggressive rules. Check `corrections.json` for blocked pairs. Add a targeted example to `verify_pspla_match` prompt.

**Roll back a bad change:**
`git stash` or dashboard Rollback button, or `git revert <hash>`.

**Deploy public site changes:**
Run generate_static.py locally (or click Publish Live in dashboard) to regenerate and encrypt `docs/index.html`, then push to GitHub. The workflow deploys to the `gh-pages` branch — direct pushes to `docs/` on main are NOT served.

**Run bulk NZSA recheck to find existing false positives:**
Dashboard → Searches → Bulk Recheck → select NZSA only → All companies → Run.
