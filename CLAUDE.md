# PSPLA Checker â€” Full Project Context for Claude Code

## What This Project Does
Automated tool that finds NZ security companies on the internet, checks whether each one holds a current PSPLA (Private Security Personnel Licensing Authority) licence, and stores results in Supabase. A Flask dashboard lets the user browse results, trigger searches, review AI decisions, and correct errors. A public GitHub Pages site shows a filtered read-only view encrypted with StatiCrypt.

Owned and operated by Wade. The goal is to build a comprehensive list of every NZ private security company and whether they are licenced.

- **Python** â€” Flask dashboard + searcher engine

## Supabase Security (RLS)

Both tables have Row Level Security enabled:
- **Companies**: `public_read_only` policy â€” anon role can SELECT, nothing else. Service role bypasses RLS entirely.
- **AuditLog**: RLS enabled with NO policies â€” fully blocked to anon. Only service role can read/write.

This means the public site (using anon key) can read Companies but cannot see AuditLog or write anything.

## Full Search Pipeline (process_and_save_company)

Every company found â€” whether via Google, Facebook, or a directory â€” passes through the same pipeline in `process_and_save_company()`:

### 1. Dedup check
- `get_company_by_name()` â€” if company name already in DB, just append the new region and return
- `company_name_exists()` â€” final dedup guard at the end before save

### 2. Google Business Profile â€” `get_google_business_profile(company_name, region)`
One targeted SerpAPI search per company. Parses `knowledge_graph` and `local_results` from SerpAPI JSON â€” returned alongside organic results at no extra quota cost. Extracts rating, reviews, phone, address. Stored in `google_rating`, `google_reviews`, `google_phone`, `google_address`.

### 3. Facebook page scrape (if `facebook_url` is known)
`scrape_facebook_page(fb_url)` â€” three-tier approach:
- **Tier 1**: parse Google/SerpAPI snippet from `_FB_SNIPPET_CACHE`
- **Tier 2**: stream first 8 KB of page HEAD to extract `og:description` meta tag
- **Tier 3**: `m.facebook.com/slug/about/` with iPhone UA + 1s delay
FB email backfilled to company email if none found elsewhere.

### 4. PSPLA check â€” `check_pspla(company_name, website_region, page_text, co_result, directors, extra_context)`
Four strategies tried in order, stopping at first hit:

**Strategy 1 â€” Name variations:** `generate_name_variations()` â€” original, hyphenâ†’space, CamelCase split, compound word split. Prefers Active licence; falls back to expired.

**Strategy 2 â€” Keyword AND search:** `extract_keywords()` strips stop words and joins first 3 keywords with AND.

**Strategy 3 â€” Single keyword:** First meaningful keyword with 6+ chars.

**Strategy 4 â€” LLM-suggested names:** Claude Haiku given ALL available context and asked to suggest up to 5 PSPLA search terms. Writes `llm_decision` audit entry.

**After finding results:**
- Multiple companies OR keyword/variant match â†’ `verify_pspla_match()` called
- Single company + full-name match â†’ trusted without verification
- Region-boosted sorting applied before verification
- Best-status doc selected: Active > Expired > Withdrawn, then most recent expiry

### 5. PSPLA verification â€” `verify_pspla_match(website_company, pspla_company, website_region, pspla_address, _audit_name)`

**Step A â€” Hard pre-check (no LLM):**
1. Forward check: every significant word (4+ chars) from website name must appear in PSPLA name. If ALL missing â†’ reject.
2. **Reverse prefix check (added 10 Mar 2026):** if the PSPLA name starts with distinctive word(s) not present in the website name, it's a different entity. E.g. website="Livewire Electrical", PSPLA="Addz Livewire Electrical" â†’ "addz" is a distinctive prefix â†’ reject. Geographic qualifiers (southern, central, northern etc.) are exempt.

**Step B â€” Lessons injection:** relevant lessons from `lessons.json` injected into prompt.

**Step C â€” Claude Sonnet verify:** Returns `{match, confidence, reason}`. Writes to AuditLog.

**Step D â€” Deep verify:** If low/medium confidence + extra context available â†’ `_llm_deep_verify()`.

**Graceful degradation:** If Claude API unavailable, returns low-confidence pass rather than false.

### 6. Companies Office check â€” `check_companies_office(company_name)`
Google search `site:companies.govt.nz`. Extracts registered name, number, NZBN, address, directors, CO status, incorporation date. Sold/re-registered detection: if status=Removed â†’ broader keyword search for successor â†’ retries PSPLA with successor name.

### 7. NZSA check â€” `check_nzsa(company_name, website=None)`
Scrapes `security.org.nz` member directory. Returns member, member_name, accredited, grade, contact_name, phone, email, overview.

**Matching logic (updated 10 Mar 2026):**
- Exact normalised name match â†’ immediate accept
- Website domain match â†’ immediate accept
- Word overlap scoring: significant word hits Ã— 3, generic word hits Ã— 1. Threshold: score â‰¥ 3.
- `_GENERIC` stop-words include: security, services, alarm, electrical, plumbing, construction, engineering, contracting, etc. Generic industry words alone do NOT constitute a significant match.
- **Domain mismatch guard:** if company website domain and NZSA member email/website domain are both company-specific (not Gmail/Hotmail etc.) and not in the directory-site exclusion list, and they differ â†’ reject. Catches "Livewire Electrical" matching "Sefton Electrical" via shared generic word.
- **Directory domain exclusion:** if company's stored URL is a directory site (moneyhub.co.nz, yellowpages.co.nz, facebook.com etc.) skip domain mismatch check â€” not the company's own domain.

### 8. Cross-check â€” `_llm_cross_check_sources()`
Claude Haiku cross-checks all source names/addresses for consistency. Informational only.

### 9. Service detection â€” `gather_service_text()` + `detect_services()`
Fetches root domain, extracts internal links matching service slugs, scrapes up to 3 sub-pages. Claude Haiku detects `has_alarm_systems`, `has_cctv_cameras`, `has_alarm_monitoring`. Also runs on Facebook data â†’ `fb_alarm_systems`, `fb_cctv_cameras`, `fb_alarm_monitoring`.

### 10. LLM association review â€” `llm_verify_associations()`
Called before saving. Rejects bad LinkedIn/FB/PSPLA associations, writes audit entries.

### 11. Individual licence check
Searches PSPLA for each director name. Sets `individual_license_found` and `license_type`.

### 12. Save â€” `save_to_supabase(record)`
HTTP POST/PATCH to Supabase REST API. Uses `RECORD_TEMPLATE` as base dict.

## The Corrections System

**Interactive correction flow (updated 10 Mar 2026):**
1. User types a correction note in the dashboard correction form
2. `POST /save-correction` â€” parses note, runs `check_pspla` to get a proposed new match, generates a lesson. Returns `{proposed: {...}}` WITHOUT saving to DB yet.
3. Dashboard shows a **Recheck Review Modal** with the proposed match details
4. User clicks "Yes, use this match" or "No, not a match"
5. `POST /confirm-recheck` â€” saves the approved/rejected result to DB

**Corrections files:**
- `corrections.json` â€” list of `{type, blocked_pspla_name, company_name, company_id, raw, summary, timestamp}`
- `lessons.json` â€” list of `{pattern_name, what_went_wrong, rule_to_apply, keywords_to_watch}`

`_is_pspla_match_blocked(company_name, pspla_name)` checked inside `check_pspla` before any match is accepted.

## User Access & Admin System

**Authentication:** Google OAuth (via Supabase Auth) or localhost password login. Google users must be in the `allowed_users` table with `active=true`.

**Admin detection** (`_is_admin()`):
1. Password login (localhost) â†’ always admin
2. `session['is_admin']` set from `allowed_users.is_admin` column during Google login
3. Email matches `NOTIFY_EMAIL` env var â†’ always admin

**Admin-only actions:** All destructive and sensitive operations are admin-only (30+ endpoints). This includes: all search launches, stop/pause/resume, rollback, delete company, clear DB, dedupe, publish, backup, export CSV, save corrections, all individual rechecks, NZSA reports, save search terms, toggle schedule, LLM log view/clear. Non-admin users can only view the main table. New users default to `is_admin=false`.

**Two-Factor Authentication (2FA):**
- Optional TOTP-based 2FA per user, compatible with Google Authenticator, Microsoft Authenticator, DUO
- Enabled via My Account â†’ Profile page
- Uses `pyotp` for TOTP generation/verification (RFC 6238, 30-second window with Â±1 tolerance)
- TOTP secrets encrypted at rest with Fernet (`TOTP_ENCRYPTION_KEY` env var, or derived from `PAGES_PASSWORD`+`SUPABASE_SERVICE_KEY`)
- 8 backup codes generated on setup (format: XXXXX-XXXXX), stored encrypted
- Login flow: Google OAuth â†’ if `totp_enabled` â†’ `session['2fa_pending']=True` â†’ redirect to `/auth/2fa` â†’ verify code â†’ complete login
- Rate limiting: 5 failed attempts â†’ session cleared, forced re-login
- **Remember device:** "Remember this device for 30 days" checkbox. Uses HMAC-signed cookie (`2fa_remember`) with `app.secret_key`. Token format: `email|expiry|signature`. Validated on next login â€” skips 2FA if valid.
- **Auto-submit:** When all 6 digits are typed, verification starts automatically (no button click needed). Uses `autoSubmitting` flag to prevent double-submit.
- Password login (localhost) bypasses 2FA entirely
- Admin can reset a user's 2FA from User Access page

**Profile & Avatar:**
- My Account page (`/account/profile`) with avatar upload and 2FA management
- Avatar stored in Supabase Storage `avatars` bucket (public)
- Upload: client reads file as base64 data URL â†’ POST to server â†’ decode â†’ upload to Supabase Storage with `x-upsert: true`
- Avatar displayed in navbar (small circle next to sign-out) and User Access page

**Access Request System:**
- Unauthorized Google users who try to login are redirected to a CAPTCHA-gated access request page
- Flow: `/auth/verify` returns `access_captcha` â†’ redirect to `/auth/request-access` â†’ user completes reCAPTCHA v2 â†’ click "Request Access" â†’ server validates CAPTCHA with Google â†’ creates `pending` record in `access_requests` â†’ sends fancy HTML email to admin
- **reCAPTCHA v2** ("I'm not a robot" checkbox): requires `RECAPTCHA_SITE_KEY` and `RECAPTCHA_SECRET_KEY` env vars. Server-side validation via `https://www.google.com/recaptcha/api/siteverify`. If keys not set, CAPTCHA widget hidden and validation skipped (graceful degradation).
- Repeat attempts while pending: shows "still pending" message, no duplicate email
- After rejection: shows soft letdown message, allows re-request (creates new pending + new email)
- Admin reviews requests on User Access page â†’ Approve (adds to `allowed_users`, sends welcome email) or Reject (with optional reason)
- Audit trail: `login_audit` records `access_requested`, `access_pending`, `access_rejected` results

**Supabase tables:**
- `allowed_users` â€” `id, email, name, added_by, added_at, active, last_login, last_provider, is_admin, totp_secret, totp_enabled, totp_backup_codes, totp_enabled_at, avatar_url`
- `login_audit` â€” `id, email, provider, result, attempted_at, user_agent`
- `access_requests` â€” `id, email, status (pending/approved/rejected), requested_at, reviewed_by, reviewed_at, reject_reason`

## Key Design Patterns

- **`RECORD_TEMPLATE`** â€” single source of truth for all DB columns. Every record starts as a copy.
- **`check_schema()`** â€” runs at startup of every search script. Verifies all RECORD_TEMPLATE keys exist. Aborts if missing.
- **`running.flag` / `pause.flag`** â€” file-based IPC. Cleared on dashboard startup and stop.
- **`search_start.json`** â€” written by `_launch()` at search start. Contains `{started, type, triggered_by}`. Used by `/search-running-info` for conflict detection.
- **`__main__` guards** on all `run_*.py` â€” prevents accidental execution when imported.
- **Orphan process kill** â€” Stop uses PowerShell WMI to kill python processes by script name.
- **`_FB_SNIPPET_CACHE`** â€” module-level dict, populated by `find_facebook_url`, consumed by `scrape_facebook_page`.
- **Sold company detection** â€” CO status=Removed â†’ successor keyword search â†’ retry PSPLA.
- **Region-boosted PSPLA sorting** â€” PSPLA names containing website region words ranked first.
- **`_LoggingAnthropicClient`** â€” intercepts all LLM calls, logs to `llm_debug.log`.
- **CSV exports** â€” always use `sorted(companies[0].keys())` for columns, never a hardcoded list.
- **Sort persistence** â€” `localStorage.setItem('pspla-sort', sel)` saves sort selection, restored on page load via IIFE.
- **Partial auth boundary** â€” `session['2fa_pending']` blocks all protected routes via `@before_request` until 2FA is completed.
- **TOTP encryption** â€” secrets encrypted with Fernet before storage; `_totp_fernet()` creates cipher from env key or derives one.
- **CSRF fetch wrapper** â€” global `fetch()` monkey-patch in `<script>` block auto-attaches CSRF token header to all non-GET requests. Forms get hidden field via `DOMContentLoaded`.
- **In-memory rate limiting** â€” `_rate_limit_store` dict with per-IP+category sliding window. No external deps (Redis). Suitable for single-instance Azure deployment.
- **Safe error logging** â€” `_safe_error(e)` logs to stderr, returns generic message to user. Never exposes paths, tracebacks, or internal details.

## Security Hardening (added 13 Mar 2026)

Comprehensive security hardening applied across the dashboard:

**CSRF Protection:**
- Session-based CSRF tokens generated via `_csrf_token()`, stored in `session['_csrf_token']`
- Token injected into all pages via `<meta name="csrf-token">` tag
- Global `fetch()` wrapper auto-attaches `X-CSRF-Token` header to all non-GET requests
- `DOMContentLoaded` listener auto-injects hidden `_csrf_token` field into all POST forms
- `_check_csrf()` validates token on all authenticated state-changing requests (POST/DELETE/PATCH/PUT)
- Auth endpoints (`_AUTH_SKIP`) exempt â€” no session exists pre-login
- Applied to: main dashboard, user access page, profile page, LLM log page, version history, search history

**Rate Limiting:**
- In-memory rate limiter (`_rate_limit_store` dict + threading lock) â€” no Redis needed for single Azure instance
- `_check_rate_limit(category, max_requests, window_seconds)` returns 429 response if exceeded
- Limits: login 5/min, auth 10/min, 2FA 5/min, access requests 3/min, search launches 3/min
- Per-IP tracking via `request.remote_addr`

**HTTP Security Headers** (`@app.after_request`):
- `X-Content-Type-Options: nosniff`
- `X-Frame-Options: SAMEORIGIN`
- `Referrer-Policy: strict-origin-when-cross-origin`
- `Permissions-Policy: camera=(), microphone=(), geolocation=()`
- `Strict-Transport-Security: max-age=31536000; includeSubDomains` (HTTPS only)
- `Content-Security-Policy`: restrictive policy allowing `'self'`, inline scripts/styles (needed for dashboard), CDNs (jsdelivr, cloudflare, Google reCAPTCHA, Google Fonts), Supabase Storage for images

**Session Cookie Hardening:**
- `SESSION_COOKIE_HTTPONLY=True` â€” no JS access to session cookie
- `SESSION_COOKIE_SAMESITE='Lax'` â€” CSRF mitigation
- `SESSION_COOKIE_SECURE=True` on production (HTTPS) â€” cookie only sent over TLS

**Input Length Validation:**
- `_validate_input_lengths(data)` checks string fields against `_INPUT_MAX_LENGTHS` dict
- Limits: correction 2000, company_name 200, email 254, name 200, phone 50, website 500, region 100, reason 500, password 128
- Applied to: `/update-company`, `/save-correction`, `/api/allowed-users` POST, `/api/access-requests/.../reject`

**Admin-Only Endpoint Guards:**
- `_is_admin()` check added to 30+ endpoints (see Admin-only actions in User Access section)
- JSON endpoints return `{"ok": false, "error": "admin only"}` with 403
- HTML/redirect endpoints show "Admin access required" flash message
- `/llm-log` and `/llm-log/clear` restricted to admin

**Error Sanitisation:**
- Global `@app.errorhandler(500)` and `@app.errorhandler(404)` â€” return safe messages, never stack traces
- `_safe_error(e, fallback)` helper logs real error to `sys.stderr` (visible in Azure Log Stream), returns generic fallback string to user
- All `str(e)` in HTTP responses replaced with `_safe_error(e)` calls
- All redirect error messages sanitised (no exception details in URL params)

**CORS:** Not configured (Flask default = same-origin policy). No `flask-cors` or `Access-Control-*` headers.

**Logging safety:** No passwords, tokens, API keys, or secrets logged. LLM debug log contains prompts/responses (AI reasoning only). Login audit logs email, provider, result, user-agent â€” no credentials.
